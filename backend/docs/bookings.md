# Модуль: bookings

Отвечает за бронирование столов в ресторане.

## Жизненный цикл бронирования

```
Пользователь создаёт бронь
        ↓
  status = "pending"  (ожидает подтверждения)
        ↓
  Менеджер в Django-админке меняет статус
        ↓
  status = "confirmed"  →  push: "Ваш столик забронирован"
  status = "canceled"   →  push: "Ваше бронирование было отменено"
        ↓ (после визита)
  status = "completed"  →  push: "Спасибо за визит!"
```

Смена статуса происходит **только через Django-админку** — публичного эндпоинта для этого нет.

## Эндпоинты

### GET /api/bookings/

Список бронирований текущего пользователя. Отсортированы по дате и времени визита (сначала самые ранние).

**Авторизация:** Bearer JWT (обязательна)

**Query-параметры:** `page`, `page_size` (по умолчанию 20, максимум 100)

**Ответ 200:**
```json
{
  "count": 3,
  "next": null,
  "previous": null,
  "results": [
    {
      "id": 7,
      "guest_name": "Алихан Сейткали",
      "date": "2026-06-15",
      "time": "19:30:00",
      "guests_count": 4,
      "comment": "Аллергия на орехи, нужен детский стул",
      "status": "confirmed",
      "created_at": "2026-05-11T10:00:00+06:00"
    }
  ]
}
```

---

### POST /api/bookings/

Создаёт новое бронирование от имени текущего пользователя.

**Авторизация:** Bearer JWT (обязательна)

**Тело запроса:**
```json
{
  "guest_name": "Алихан Сейткали",
  "date": "2026-06-15",
  "time": "19:30:00",
  "guests_count": 4,
  "comment": "Аллергия на орехи, нужен детский стул"
}
```

| Поле | Обязательное | Ограничения |
|---|---|---|
| `guest_name` | Да | до 255 символов |
| `date` | Да | формат `YYYY-MM-DD` |
| `time` | Да | формат `HH:MM:SS` |
| `guests_count` | Да | от 1 до 50 |
| `comment` | Нет | произвольный текст |

**Ответ 201:**
```json
{
  "id": 8,
  "guest_name": "Алихан Сейткали",
  "date": "2026-06-15",
  "time": "19:30:00",
  "guests_count": 4,
  "comment": "Аллергия на орехи, нужен детский стул",
  "status": "pending",
  "created_at": "2026-05-11T10:05:00+06:00"
}
```

Поля `id`, `status` и `created_at` — только для чтения, клиент их не передаёт.

**Ответ 400** — примеры ошибок:
```json
{ "guests_count": ["Убедитесь, что это значение меньше либо равно 50."] }
{ "guest_name": ["Обязательное поле."] }
```

## Модель TableBooking

| Поле | Тип | Описание |
|---|---|---|
| `id` | int | Первичный ключ |
| `user` | FK → User | Владелец брони (null допустим — для броней от администратора) |
| `guest_name` | string | Имя гостя (может отличаться от профиля) |
| `date` | date | Дата визита |
| `time` | time | Время визита |
| `guests_count` | int | Количество гостей (1–50) |
| `comment` | text | Дополнительные пожелания (необязательно) |
| `status` | string | Статус (см. ниже) |
| `created_at` | datetime | Дата создания |
| `updated_at` | datetime | Дата последнего изменения |

### Статусы бронирования

| Значение | Описание |
|---|---|
| `pending` | Ожидает подтверждения (начальный статус) |
| `confirmed` | Подтверждено менеджером |
| `canceled` | Отменено |
| `completed` | Визит состоялся |

## Push-уведомления

Уведомления уходят автоматически при смене статуса через Django-сигнал (`signals.py`):

| Новый статус | Заголовок | Текст |
|---|---|---|
| `confirmed` | Бронирование подтверждено | Ваш столик забронирован. Ждём вас! |
| `canceled` | Бронирование отменено | Ваше бронирование было отменено. |
| `completed` | Спасибо за визит! | Рады были вас видеть. Ждём снова! |

Push уходит только если у пользователя есть зарегистрированное FCM-устройство.

## Права доступа в Django-админке

При первой миграции автоматически создаётся группа **«Менеджер зала»** с правами:
- `view_tablebooking` — просмотр броней
- `change_tablebooking` — изменение статуса

Добавить/удалить бронирование менеджер не может.

## Важные нюансы

- `user` в модели — `null=True`. Это сделано намеренно: администратор может создать бронь для гостя без аккаунта.
- Пользователь через API видит **только свои** бронирования — фильтрация `filter(user=request.user)` в `get_queryset`.
- `_original_status` сохраняется в `__init__` модели — это нужно сигналу, чтобы понять, изменился ли статус.
- Нет ограничения на бронирование в прошлом — валидацию дат при необходимости нужно добавить в сериализатор.

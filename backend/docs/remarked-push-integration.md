# Интеграция push-уведомлений для Remarked

Документ для передачи Remarked при настройке прямой отправки push-уведомлений
в приложение PILIGRIM через Firebase Cloud Messaging (FCM), в обход нашего
бэкенда. Описывает формат сообщения и что реально умеет обрабатывать
приложение — остальное (сама отправка через Firebase HTTP v1 API) описано в
официальной документации Firebase.

## Доступ

Отдельный service account с ролью **Firebase Cloud Messaging API Admin**
(`roles/firebasecloudmessaging.admin`) — доступ только на отправку push,
без доступа к остальным сервисам Firebase-проекта (Firestore, Auth, Storage).
JSON-ключ передаётся отдельно.

Официальная документация по отправке через этот ключ:
https://firebase.google.com/docs/cloud-messaging/send-message

## На какой токен отправлять

Токен устройства гостя (`firebase_token`) уже передаётся вам при регистрации/
входе гостя через `POST /store/customer/create` (Mobile API v2) — отдельного
запроса на получение токена не требуется, он привязан к гостю в вашей системе.

Если у гостя несколько устройств — актуальным считается последний переданный
`firebase_token` (при повторном логине на новом устройстве старый токен
перезаписывается).

## Формат сообщения

```json
{
  "message": {
    "token": "<firebase_token гостя>",
    "notification": {
      "title": "Заголовок уведомления",
      "body": "Текст уведомления"
    },
    "data": {
      "type": "booking"
    }
  }
}
```

**Требование Firebase (не наше):** все значения в `data` должны быть
строками — не числами, не булевыми, не вложенными объектами.

## Поле `data.type` — deep-link при тапе на уведомление

Это единственное поле в `data`, которое приложение сейчас читает и
обрабатывает. Поддерживаемые значения:

| `type` | Что делает приложение |
|---|---|
| `booking` | Открывает экран бронирования столика |
| `event` | Переключает на вкладку «Мероприятия» |
| любое другое / отсутствует | Ничего — уведомление просто показывается, тап никуда не ведёт |

Если нужен другой сценарий (например, диплинк на конкретное блюдо/акцию) —
это отдельная доработка на нашей стороне, сейчас такого нет. Сообщите
заранее, что именно нужно, чтобы мы это добавили до начала использования.

Остальные поля в `data` (кроме `type`) можно добавлять свободно — приложение
их просто проигнорирует, ошибки не будет.

## Поведение в приложении

- **Приложение открыто (foreground):** показывается всплывающий тост с
  `title`/`body` из `notification`. `data` в этот момент не читается вообще
  (используется только при тапе на системное уведомление).
- **Приложение свёрнуто/закрыто:** стандартное системное уведомление
  Android/iOS; при тапе — запускается приложение и обрабатывается
  `data.type` (см. таблицу выше).

## Важно: при прямой отправке через Firebase не действуют наши ограничения

Сейчас все push, которые уходят через **наш** бэкенд, проходят через:
- **Тихие часы** — push с категорией (`event`/`promotions`/`closed_events`)
  не отправляются вне 09:00–21:00 (по местному времени точки), откладываются
  до утра;
- **Недельный лимит** — не больше 3 push с категорией на одного гостя в
  неделю;
- Внутренний аудит рассылок (кто когда что отправил).

Если вы отправляете push **напрямую через Firebase** (минуя наш бэкенд) —
**ничего из этого не действует**. Ограничения по частоте/времени отправки
в этом случае — полностью на вашей стороне.

## Регистрация устройства (справочно)

Помимо `firebase_token`, синхронизируемого через `/store/customer/create`,
в приложении есть собственная регистрация устройств
(`POST /api/v1/notifications/device/register/`, модель `UserDevice`) — это
наш внутренний канал для push, инициированных нашим бэкендом (статусы
брони, ручные рассылки через панель/бот), и он никак не пересекается с
вашей интеграцией. Знать о нём для настройки со стороны Remarked не нужно,
упомянуто для полноты картины.

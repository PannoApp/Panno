Этот документ содержит детальный анализ несоответствий текущей кодовой базы (Flutter-клиент и Django-бэкенд) требованиям «ТЗ ДЛЯ ПИЛИГРИМ» и спецификации API, а также предлагает план доработок.

В соответствии с правилом @rules:Не меняй код, все недочеты выпиши в ПЛАН, изменения в коде на данном этапе не вносились.

1. Раздел «Интерьер и 3D-тур» (ТЗ, Раздел 4.3)
Текущая реализация экрана интерьера (lib/screens/interior_screen.dart) представляет собой простую сетку изображений и содержит критические расхождения с требованиями ТЗ:

Отсутствие 3D-тура:
Требование ТЗ: Отображение WebView для виртуального 3D-тура по заведению (сферические панорамы). Если 3D-тур не настроен в панели управления (tour_link == null в API GET /api/v1/core/info/), отображается обычная фотогалерея.
Факт в коде: Поле tour_link полностью игнорируется. WebView или кнопка перехода к 3D-туру на экране InteriorScreen отсутствуют.
Отсутствие интерактивности и зума:
Требование ТЗ: При тапе на фото открывается полноэкранный режим с поддержкой Zoom (например, через InteractiveViewer) и текстовым описанием (информационные точки/подсказки с сервера).
Факт в коде: Элементы сетки картинок не реагируют на нажатия. Просмотр в полный экран, зум и отображение подписей к фото (caption / zone_display) не реализованы.
Отсутствие группировки по зонам:
Требование ТЗ: Фотогалерея сгруппирована по зонам (например: «Основной зал», «Терраса», «VIP-комната»).
Факт в коде: Фотографии выводятся сплошным списком в виде плоской сетки без визуального разделения или вкладок по зонам.
Отсутствие атмосферного аудио:
Требование ТЗ: Фоновое воспроизведение легкого эмбиента или звуков степи (ветер, потрескивание костра) при просмотре галереи с возможностью отключения.
Факт в коде: Фоновое аудио полностью отсутствует.
2. Раздел «Афиша и новости» (ТЗ, Раздел 4.4)
Отсутствие фотоотчетов для прошедших мероприятий:
Требование ТЗ: Должен быть доступен архив прошедших мероприятий с возможностью просмотра фотоотчетов.
Факт в коде:
Бэкенд: В моделях Django (Event в backend/apps/events/models.py) и сериализаторах отсутствует связь с фотографиями фотоотчетов или массив изображений. Нет API эндпоинтов для загрузки/получения фотоотчетов.
Фронтенд: В модели ApiEvent во Flutter есть поле hasPhotoReport, однако в UI экрана деталей события EventDetailScreen при event.isPast = true кнопка «Записаться» просто скрывается. Никакой галереи или отображения фотографий отчета нет.
Разграничение доступа и стоимость:
Требование ТЗ: Для закрытых мероприятий (format == "closed") должен отображаться статус особого доступа.
Факт в коде: Закрытые мероприятия отображаются обычным текстовым чипом, нет логики предупреждения пользователя о VIP/закрытом статусе до начала записи.
3. Раздел «Бронирование столов» (ТЗ, Раздел 4.5)
Логика депозита (Оплата и звонок менеджеру):
Требование ТЗ: Если для бронирования требуется депозит (bookingDepositRequired == true в API GET /api/v1/core/info/), приложение должно отобразить баннер/диалог с текстом bookingDepositNote и кнопкой звонка менеджеру (tel: ссылка на номер телефона заведения). Оплата через приложение не производится — только звонок.
Факт в коде: В lib/screens/booking_screen.dart при необходимости депозита выводится только статический текстовый блок предупреждения. Кнопка «Позвонить менеджеру» отсутствует, что вынуждает пользователя вручную искать контакты.
Отсутствие полноценного экрана успешного бронирования (Success Screen):
Требование ТЗ: После успешной отправки заявки пользователь должен четко видеть статус и дальнейшие шаги.
Факт в коде: При успешном создании бронирования (_submitted == true) внизу формы просто дорисовывается карточка _FlowCard со сценарием. Сама форма бронирования остается активной и заполненной. Это нарушает UX (пользователь может нажать кнопку отправки еще раз) и выглядит неаккуратно. Необходим переход на выделенный экран успеха бронирования с анимацией и кнопками перехода в историю броней или на главную.
4. Логика фильтрации по тегам и категорий в меню (Menu)
Статус поддержки категорий (Горячие, Бар, Винная карта):
Статус: Полностью реализовано и корректно работает.
Логика: При переходе на классический вид меню приложение выполняет запрос к эндпоинту GET /api/v1/menu/categories/ для получения списка категорий. Данные парсятся в модели ApiCategory и отображаются в виде горизонтальной панели вкладок _CategoryTabs (первая вкладка — «Все блюда», далее идут динамические категории с сервера: «Горячие», «Бар», «Винная карта» и т.д.). Выбор вкладки отправляет запрос на сервер с фильтром по category_id для соответствующей категории.
Клиентская фильтрация по тегам при серверной пагинации (Критический баг):
Факт в коде: В lib/screens/menu_screen.dart фильтрация по тегам (_FilterChips) реализована локально на клиенте в методе _applyTagFilter(menuProvider.dishes).
В чем баг: Меню запрашивается с сервера постранично (страницами по N элементов). Если пользователь выбирает фильтр-чип тега (например, «Острые»), клиент фильтрует только локальный список загруженных на данный момент блюд. Если на первой странице нет острых блюд, приложение покажет пустой экран, хотя на сервере на последующих страницах такие блюда есть.
Решение: Перенести фильтрацию по тегам на бэкенд. Метод fetchDishes в MenuRepository уже поддерживает параметр tag_ids, но MenuProvider его не использует. Нужно передавать выбранные теги в API.
5. Идемпотентность и сетевая устойчивость (Resiliency)
Генерация ключей идемпотентности:
Требование ТЗ/API: Все POST запросы на создание бронирований и записей на мероприятия требуют заголовок Idempotency-Key (UUID v4) для предотвращения дубликатов при нестабильной связи.
Факт в коде: UUID генерируется прямо в репозитории на каждый вызов метода. Если запрос упадет из-за таймаута сети (при этом сервер мог успешно создать бронь), повторное нажатие кнопки пользователем сгенерирует новый UUID, что приведет к созданию второй (дублирующей) брони на сервере.
Решение: Ключ идемпотентности должен генерироваться на уровне экрана/презентера при начале отправки формы и сохраняться до тех пор, пока запрос не завершится успехом или пользователь явно не сбросит форму. При повторных попытках отправки (retry) должен использоваться тот же самый UUID.
6. Терминология («Герои» вместо «Гостей»)
Требуется провести финальный аудит всех строковых ресурсов во Flutter-проекте и заменить оставшиеся вхождения корня «гост» (гости, гостей, гостя) на «герой» (герои, героев, героя) в контексте пользователей приложения, за исключением системных терминов бэкенда (например, полей API guests_count).

Шаг 5. Идемпотентность
Вынести генерацию Idempotency-Key на уровень BookingProvider / EventsProvider. Сохранять сгенерированный UUID во время выполнения запроса и переиспользовать его при повторных попытках отправки одной и той же формы.
